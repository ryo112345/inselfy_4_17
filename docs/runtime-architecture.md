# ランタイム構成 — 1コンテナ2プロセスの設計判断

## 構成

Cloud Run の 1 サービス・1 コンテナに Next.js（port 8080）と Go API（port 8081）を
同居させ、`docker/start.sh` が両プロセスを起動・監視する。

```
インターネット → Cloud Run (port 8080)
                   └─ Next.js ─ SSR / 静的配信
                        └─ /api/* を http://127.0.0.1:8081 へプロキシ → Go API → Neon
```

## なぜサイドカーや別サービスに分離しないのか

**コストと運用の最適化を意図した判断**であり、分離パターンを知らないからではない。

- **無料枠・低コスト運用が最優先**（GCP 月予算 ¥500）。Cloud Run を 2 サービスに
  分けると、リクエスト課金・min-instances・監視対象がそれぞれ倍になる
- **サービス間通信の問題が消える**: front → API が localhost なので、
  サービス間の認証（IAM invoker）・ネットワークレイテンシ・egress を考えなくてよい
- **デプロイの原子性**: front と API のバージョンが常に一致する。
  スキーマファースト開発（OpenAPI 契約から両側を生成）と相性がよく、
  「front だけ新しくて API が古い」状態が構造的に発生しない

## トレードオフと緩和策

| リスク | 緩和策 |
|---|---|
| 片方のプロセスだけ死ぬと Cloud Run から見えない | `start.sh` が両プロセスを監視し、どちらかが死んだらコンテナごと終了 → Cloud Run が再起動（詳細は start.sh のコメント） |
| probe が Next.js にしか届かない | front の catch-all プロキシ経由で `/api/healthz` `/api/readyz` を probe（Go API まで貫通） |
| front と API を独立にスケールできない | 現状のトラフィックでは不要。必要になったら分離する（下記） |
| コンテナ突破時の被害 | 非 root（node ユーザー）で実行。書き込み可能なのは uploads と Next.js キャッシュのみ |

## 分離すべきタイミング

以下のいずれかが起きたら 2 サービス化を検討する:

- front と API の負荷特性が乖離し、独立したスケーリング設定が必要になった
- API のデプロイ頻度が front と大きくずれ、片側だけのリリースが常態化した
- WebSocket 接続数が増え、API 側だけ長い timeout / 常駐インスタンスが必要になった
