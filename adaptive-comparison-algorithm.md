# アダプティブ一対比較アルゴリズム設計

## 1. 概要

21 Work Needs の一対比較（全210ペア）を、アダプティブ方式により最大70問で実施する。
フロントエンド側で Bradley-Terry モデルをリアルタイム推定し、次のペアを動的に選択する。

**診断の目的:** 21ニーズ全体の精密なランキングではなく、**上位3つ（重視するもの）と下位3つ（重視しないもの）の特定**を主目的とする。70問という制約上、中間ランクの精密な序列化は行わない。

---

## 2. アルゴリズム全体フロー

```
初期化: μ = [0, ..., 0] (21次元), 比較行列 Y = 空
        未出題ペアプール = 210ペア全件

for q = 1, 2, ..., 70:
  1. 次のペアを選択（セクション3参照）
  2. ユーザーに提示し、回答を受け取る
  3. 比較行列 Y を更新
  4. BTモデルを再推定 → μ, SE を更新
  5. 70問に達したら終了
```

---

## 3. ペア選択戦略

### 3.1 フェーズ1: 初期ラウンドロビン（1〜21問目）

最初の21問は推定が不安定なため、全ニーズを均等にカバーする。

**サーバー側で事前に21ペアを生成**し、セッション開始時にフロントへ返す。

生成方法（21頂点の巡回グラフ: 各ニーズがちょうど2回出現）:

```
21ニーズをランダムにシャッフルし、隣接ペアで巡回:
(need_1, need_2), (need_2, need_3), ..., (need_20, need_21), (need_21, need_1)
```

提示順もシャッフルする。この方式で全21ニーズが正確に2回ずつ出現することが保証される。

**セッション開始フロー:**

```
1. フロント → サーバー: セッション開始リクエスト
2. サーバー: 21ペアを生成し、session_id と紐づけて保存
3. サーバー → フロント: session_id + 初期21ペア
4. フロント: 21問を出題しながらBT推定開始
5. フロント: 22問目以降はアダプティブに選択（フロント側で計算）
```

### 3.2 フェーズ2: 境界確定フェーズ（22問目〜）

上位3・下位3を確定させることを優先するペア選択を行う。

**ステップ1: 現在のランキングから境界候補を特定**

μ の降順で並べたとき:
- 上位境界: 3位と4位の間（上位グループに入るかどうか）
- 下位境界: 18位と19位の間（下位グループに入るかどうか）

**ステップ2: 境界付近のペアを優先**

```
score(i, j) = I(i, j) * boundary_weight(i, j)
```

Fisher情報量:
```
I(i, j) = p_ij * (1 - p_ij)
p_ij = 1 / (1 + exp(-(μ_i - μ_j)))
```

境界加重:
```
boundary_weight(i, j) =
  3.0  ... i または j が境界候補（ランク 2〜5位 or 17〜20位）
  1.5  ... i または j が上位/下位グループ内（ランク 1位 or 21位）
  1.0  ... それ以外（中間グループ同士）
```

境界候補同士のペアや、境界候補と隣接ランクのペアが優先的に選ばれる。
中間グループ同士のペアも完全には排除せず、全体の推定安定性を維持する。

### 3.3 出現回数制約

各ニーズの全体での出現回数に上下限を設ける（平均: 140回 / 21ニーズ ≈ 6.7回）:

- **上限: 12回** — 上限に達したニーズを含むペアはスコア0とし、選択対象から除外する
- **下限: 5回** — 残り質問数的に下限を割るニーズが出そうな場合、そのニーズを含むペアを優先する

下限チェック:

```
残り質問数 = 70 - 現在の問番号
不足ニーズ = [出現回数 < 5 かつ 残り質問数で5回に届かなくなるニーズ]
不足ニーズがある場合、そのニーズを含むペアを最優先で選択する
```

### 3.4 連続出現制約

選択されたペアが以下の制約に違反する場合、次点のペアを選択する:

- 直近3問以内に同一ニーズが2回以上出現しない

---

## 4. リアルタイムBT推定

### 4.1 推定タイミング

各回答後に即座にBTモデルを再推定する。

### 4.2 推定方法

CLAUDE.md Phase 2 のBTモデルと同一:

```
目的関数: ℓ(μ) = Σ [Y[i][j] * log(p_ij) + Y[j][i] * log(1 - p_ij)] - (1/2σ²) * Σμ_i²
```

- L2正則化: σ² = 3.0
- 識別制約: Σμ_i = 0
- 最適化: Newton-Raphson法（21次元なので十分高速）
- 収束条件: 勾配ノルム < 1e-6 or 最大100反復

### 4.3 SE（標準誤差）の計算

ヘッセ行列 H の逆行列の対角要素から算出:

```
SE_i = sqrt(diag(H⁻¹)_i)
```

ヘッセ行列の各要素:

```
H[i][i] = Σ_{j≠i} p_ij * (1 - p_ij) + 1/σ²
H[i][j] = -p_ij * (1 - p_ij)    (i ≠ j, ペア (i,j) が出題済みの場合)
H[i][j] = 0                      (未出題ペア)
```

---

## 5. 終了条件

固定70問で終了する。早期終了は行わない。

70問の一対比較により、上位3つ・下位3つのグループ間の勝率は約89〜92%に達し、十分な識別精度が得られる。

---

## 6. 整合性チェック（不完全データ対応）

70問では210ペア中の一部しか出題されないため、整合性係数の計算を不完全データに対応させる。

### 6.1 計算方法

全 21C3 = 1,330 個の三つ組のうち、**3辺すべてが出題済みの三つ組のみ**を対象とする:

```
対象三つ組数 T = |{(i,j,k) : (i,j), (j,k), (i,k) すべて出題済み}|
循環三つ組数 d = |{(i,j,k) ∈ T : A>B, B>C, C>A の循環}|
d_max_T = floor(T / 4)  (対象三つ組数に対する理論最大循環数)
ζ = 1 - d / d_max_T
```

### 6.2 最低三つ組数

対象三つ組数 T が30未満の場合、整合性係数は算出せず「データ不足」とする。

### 6.3 判定基準

CLAUDE.md と同一の閾値を使用:

| ζ | 判定 |
|---|------|
| >= 0.75 | 高い |
| 0.50-0.74 | 中程度 |
| 0.30-0.49 | 低い |
| < 0.30 | 非常に低い |

---

## 7. 結果の解釈と表示

### 7.1 グループ分類

全21ニーズの精密なランキングではなく、3グループに分類して表示する:

- **重視グループ（上位）**: μ が上位3位以内で、4位との差が統計的に有意
- **非重視グループ（下位）**: μ が下位3位以内で、18位との差が統計的に有意
- **中間グループ**: それ以外

### 7.2 境界が不明確な場合

70問で上位/下位の境界が確定しなかった場合（SE が大きく重複がある場合）:
- 境界付近のニーズは「重視の可能性がある」「非重視の可能性がある」として表示
- グループのサイズは3に固定せず、確信度に応じて2〜4の幅を許容する

---

## 8. エッジケース

| ケース | 対応 |
|--------|------|
| 特定ニーズが全勝/全敗 | L2正則化により発散しない。SE は高めに出るため追加質問で優先される |
| 70問で境界が確定しない | 結果には SE 情報を付与し、境界が不明確なことを明示 |
| 全ニーズの μ が均一（分化が低い） | 正常な結果。「明確な重視/非重視がない」という診断結果として表示 |
| 回答が極端に矛盾（整合性が低い） | 整合性警告を付与。再回答を推奨するが結果は表示する |

---

## 9. フロントエンド実装上の注意

- BT推定（Newton-Raphson）は21次元なので、ブラウザ上でもミリ秒単位で完了する
- 各回答後の再推定 + 次ペア選択は体感遅延なし
- 推定状態（μ, Y, 出題済みペア）はメモリ上に保持し、ページリロード対策として sessionStorage にも保存する
- 最終結果送信時は全回答データ（responses）とBT推定結果（μ, SE）の両方をサーバーに送る
- サーバー側の検証:
  1. 初期21ペアの回答がすべて含まれているか確認
  2. responses から BT推定を再計算し、フロントから送られた μ と一致するか検証
  3. 一致しない場合は不正な送信として拒否する（許容誤差: 各 μ の差が 1e-3 以内）
